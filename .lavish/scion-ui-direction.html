02 / 三态深度规格

共享一套骨架，各自解决不同的判断问题

三条并行分支已经统一：运行态解释过程，审批态解释后果，完成态解释证据。

运行中process

首屏层级

  • 目标与站点
  • 唯一实时状态行
  • 最近三步；更早步骤折叠
  • 暂停 / 停止只出现一次
  • 连续控制输入固定在底部

状态变化

  • 观察 → 准备 → 执行 → 核对 → 恢复
  • 无可信总步数时不显示百分比或 x/x
  • 只有 observed 才显示已完成

动效

  • 文案 140–180ms 交叉淡化
  • 新步骤 180ms / 6px 入场
  • 只有当前动作显示 900ms 单圈活动指示
契约缺口需要结构化 TaskActivity phase、phaseStartedAt、lastActivityAt；现阶段可由最新 attempt 保守推导。

03 / 证据基线

设计不能脱离真实状态截图

后续每一次实现都应在相同宽度、相同状态下做前后对照，而不是只验证页面能否打开。

04 / 交互规则

五条规则决定这套界面是否可信

这些是后续组件、文案和动效共同遵守的约束，不是视觉装饰规范。

目标始终可见

无论处于哪个状态，用户都能一眼确认“它正在替我做什么”。

只强调当前动作

历史步骤自动降权，运行中只有“现在”拥有动效和最高对比度。

审批说明后果

显示对象、站点、影响与一次性边界，不展示泛化的内部动作文本。

证据先于完成

模型说 done 不改变 UI；只有页面证据通过后才进入绿色完成态。

状态决定密度

运行时展开、审批时聚焦、完成后压缩，让侧栏不会无限变长。

收藏不是资产，被项目验证过的机制才是

Anime.js 是第一份机制地图；Scion 会成为第一轮“研究 → 采用 → 验证”的闭环。
