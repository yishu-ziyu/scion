# 001 — Nanobrowser 二开：可验证浏览器行动 Agent PRD

## 文档状态

本 PRD 定义第一个交付周期的产品范围与验收口径。产品方向、Chrome 扩展载体和核心闭环已经确定；代码处于开发与真实浏览器验收阶段，Owner 确认本文后可将状态改为 `current`。

| 项目 | 内容 |
|---|---|
| 产品 | Nanobrowser 二开 |
| 版本 | 第一个交付周期 |
| 产品类型 | 单用户、个人使用的浏览器行动 Agent |
| 主要用户 | Owner；需要在已登录网页中委托多步操作的浏览器重度用户 |
| 运行载体 | 用户日常使用的 Chrome 与现有扩展侧边栏 |
| 核心场景 | 表单处理、跨页面操作、媒体控制、成功任务复用 |

## 一句话定义

用户在日常 Chrome 中用自然语言委托多步网页操作；Agent 持续行动、接受后续控制，并且只有在取得新的可观察证据后才能宣告完成。

## 背景与机会

现有 Nanobrowser 已具备侧边栏、网页观察、规划与执行、多种浏览器动作、模型配置和本地存储，但“能够点击”还不等于“值得委托”。用户真正需要的是一份可信契约：任务不会停在中间步骤，不会丢失“这个视频”等上下文，不会只凭模型自述成功，也不会在未经同意时提交表单、发送消息或产生其他外部结果。

本次二开的价值不是再做一个侧边栏聊天助手，而是把现有自动化能力收束为完整闭环：

```text
提出目标
→ Agent 持续操作
→ 必要时请求批准或用户接管
→ 用页面状态验证结果
→ 生成完成回执
→ 可选保存为本地 Skill
```

## 用户与任务

### 目标用户

- 日常工作集中在 Chrome 已登录网页中的个人用户。
- 愿意自行配置模型与密钥，希望数据和登录态留在本地浏览器中。
- 有重复、多步骤或需要跨页面完成的网页操作，不满足于“帮我打开页面”。
- 希望保留最终控制权，但不愿逐步确认每一次普通点击。

### 核心待办

当我需要在网页中完成一段多步骤工作时，我希望只说明目标和必要输入，Agent 能利用当前登录态把事情做完、在关键动作前让我确认，并给出可信的完成证据，使我无需盯着每一步操作。

### 首批真实场景

1. 飞书表单：进入正确表单、填写字段、在提交前请求批准、提交后验证成功状态。
2. B 站媒体：进入收藏夹、播放目标视频；用户随后说“暂停这个视频”时，继续控制同一标签页与媒体对象，并验证已经暂停。
3. Skill 复用：把一次成功表单任务保存为本地 Skill，更换输入并在页面元素顺序变化后再次完成。

## 问题定义

当前浏览器 Agent 缺少四项委托保证：

1. 完整执行：可能只完成导航或中间步骤，用户仍需接手收尾。
2. 连续上下文：后续指令可能丢失原任务、标签页或页面对象。
3. 可验证完成：模型可以给出看似合理的“已完成”，但页面结果并未发生。
4. 安全复用：历史动作回放依赖旧元素位置，任意脚本又扩大安全边界。

这些问题直接破坏信任。错误的成功声明会让用户误以为工作已经完成；错误的提交、购买、发送或删除则会产生真实外部影响。

## 产品目标

### 本周期必须实现

1. 在真实 Chrome 登录态中完成一段包含导航、输入、点击和状态变化的多步任务。
2. 将暂停、继续、修改等后续指令绑定到同一任务及其浏览器目标。
3. 通过页面可观察结果判断成功，并向用户展示完成回执。
4. 所有外部提交在执行前经过一次性明确批准。
5. 任务在侧边栏重连或扩展进程中断后保留语义状态，绝不静默重复状态不明的提交。
6. 将已验证任务保存为本地 Skill，并使用新输入重新规划执行。

### 明确不做

- 不新造浏览器，不 fork Chromium，不迁移到云端浏览器。
- 不支持 Chrome 关闭后的无人值守执行。
- 不支持并行活动任务。
- 不做 Skill 分享、社区、市场或远程同步。
- 不做企业组织、角色权限、计费或运营后台。
- 不控制原生桌面应用，不自动输入凭证，不绕过验证码。
- 不执行模型生成的任意 JavaScript。
- 不以飞书、B 站专用代码伪装成通用能力。
- 不保留或恢复依赖旧元素索引的原始动作回放。

## 产品原则

1. 结果优先：打开页面或执行过动作不等于完成目标。
2. 证据优先：模型可以建议完成，运行时才拥有最终完成权。
3. 关键处批准：普通读取和可逆动作自动执行，外部提交必须明确批准。
4. 语义复用：Skill 保存目标、输入和完成契约，不保存旧页面坐标或元素索引。
5. 本地优先：复用用户现有 Chrome 登录态，任务状态与 Skill 保存在本地。
6. 通用优先：能力以站点无关的动作和观察表达，不为首批验收网站写专用流程。
7. 人类可读：界面只呈现“你 / 助手”、当前状态和下一步，不暴露内部 Agent 角色或机器状态枚举。

## 核心概念

| 概念 | 产品含义 |
|---|---|
| Task | 一个用户目标及其浏览器上下文、状态、动作、批准和完成条件 |
| Task Round | Task 中由一次新指令或后续指令产生的执行轮次；每轮拥有独立且不可变的完成回执 |
| 浏览器目标 | 当前任务引用的标签页、页面或媒体对象；重新观察后可重新绑定，不依赖旧元素索引 |
| 完成条件 | 执行前冻结的可观察结果，包含目标、基线、时效和判定方式 |
| 完成回执 | 记录本轮结果、证据、外部提交、目标与时间的不可变摘要 |
| 外部提交 | 会向外部系统产生可见或难以撤销结果的动作，例如提交、购买、发送、发布、删除或权限变更 |
| Skill | 可复用的语义任务模板，包含输入、预期结果和审批策略 |

## 核心用户流程

### 流程 A：提交表单

1. 用户说明目标并提供普通字段值。
2. 系统创建 Task 和首个 Task Round，绑定当前标签页。
3. Agent 导航、定位表单并填写字段。
4. 系统识别提交为外部提交，停止本批剩余动作并展示批准卡片。
5. 用户批准后，系统确认页面未变化并仅执行这一次提交。
6. 系统重新观察页面，检查成功文字、网址或元素状态。
7. 条件全部通过后展示完成回执；失败则继续处理或请求用户介入。
8. 用户可将该任务保存为本地 Skill。

### 流程 B：连续媒体控制

1. 用户要求进入收藏夹并播放目标视频。
2. Agent 导航、选择视频、开始播放并记录标签页和媒体目标。
3. 用户说“暂停这个视频”。
4. 系统在同一 Task 中创建新 Task Round，把“这个视频”解析为最近的有效媒体目标。
5. Agent 暂停媒体，并以页面媒体状态 `paused` 验证结果。

### 流程 C：运行本地 Skill

1. 用户从本地列表选择一个 Skill。
2. 系统只询问该 Skill 声明的输入。
3. 系统创建新 Task，按当前页面重新规划，不使用上次元素索引。
4. 运行仍遵循相同的审批和完成验证规则。
5. 若扩展冷中断，输入不从持久化状态恢复，用户需重新输入并开始新任务。

## 功能需求

### PR-01 任务创建与完整执行

- 每条新目标创建一个持久化 Task 和首个 Task Round。
- 本周期只允许一个活动任务；已有活动任务时拒绝新任务并说明原因。
- Agent 持续行动，直到本轮进入 `已完成`、`等待批准`、`等待用户`、`已暂停`、`已中断`、`失败` 或 `已取消`。
- 单纯打开页面、执行动作或模型输出“完成”均不是成功条件。

### PR-02 后续指令与连续控制

- 暂停、继续、取消、批准、拒绝和普通后续指令都必须作用于明确的 Task。
- 每条后续指令创建新 Task Round，不修改上一轮回执。
- 系统优先解析仍存在且网址合理的最近目标，其次使用 Task 内同类型目标；无法消歧时请求用户重新绑定。
- 重复命令只执行一次；过期轮次或过期状态的命令必须拒绝。

### PR-03 状态与进度

- 侧边栏必须展示当前目标、状态、最近一个有意义动作、当前网站或标签页，以及唯一明确的下一步。
- 运行中最多展示一条实时进度，不把内部 Planner、Navigator、Validator 或原始错误枚举作为用户文案。
- 可恢复失败应提供重试或改写入口；登录、验证码和目标丢失应明确要求用户处理。
- 状态不能只依赖颜色表达；批准、拒绝、继续和取消必须有清晰文字按钮。

### PR-04 完成条件

- 每轮至少声明一个完成条件；外部提交前必须冻结条件。
- 条件必须绑定当前 Task Round、浏览器目标、执行前基线、最早生效时间和超时时间。
- 首个周期支持：网址相等、前缀匹配或变化；可见文字出现或消失；交互元素状态或值；网页媒体播放或暂停；用户针对具体条件的明确确认。
- 状态改变型任务必须由动作后的新证据证明；执行前已经成立的弱条件不能证明完成。
- 普通聊天文本不能代替专用的用户完成确认。

### PR-05 完成回执

- 只有全部必需条件通过后，Task Round 才能进入 `已完成`。
- 回执至少包含结果摘要、逐项条件与观测证据、已执行的外部提交、目标网址或标签页及时间。
- 回执一旦生成不得被后续指令修改；同一 Task 的后续操作生成新回执。
- 条件可检查但失败时继续执行；无法检查时进入 `等待用户`，不得虚报成功。

### PR-06 动作风险与批准

| 风险级别 | 例子 | 默认行为 |
|---|---|---|
| 读取 | 导航、滚动、观察 | 在网址策略内自动执行 |
| 可逆 | 输入、打开普通标签页、媒体控制 | 自动执行并记录脱敏摘要 |
| 外部提交 | 提交、购买、发送、发布、删除、权限变更 | 执行前一次性批准 |
| 禁止 | 输入凭证、绕过验证码、提取秘密、越过安全网址 | 阻止或交还用户 |

- 单动作、多动作、点击和按回车提交必须经过同一风险判断入口。
- 批准只对当前任务、轮次、动作和页面状态生效一次，不形成长期授权。
- 用户批准后页面已经变化时，原批准失效，系统重新规划。
- 风险无法确定时按外部提交处理。

### PR-07 中断与恢复

- Task 元数据、轮次、状态、目标引用、批准、脱敏动作记录、证据和回执保存在本地。
- 侧边栏重连或扩展进程重启后，系统必须先重新观察页面，再允许用户显式恢复。
- 普通动作中断标记为 `已中断`；外部提交已经开始但尚无结果证据时标记为 `结果不确定 / 等待用户`。
- 结果不确定的外部提交绝不自动重试。
- 目标标签页消失时，系统说明丢失的目标并允许用户将活动标签页重新绑定。

### PR-08 本地 Skill

- 只有拥有有效完成回执的 Task Round 才能保存为 Skill。
- Skill 保存名称、语义指令模板、声明输入、完成条件、审批策略、来源任务和版本。
- Skill 不保存凭证、用户上次输入值、页面正文、旧元素索引或任意脚本。
- 运行 Skill 时校验输入完整性，并以当前页面重新规划。
- 首个周期支持查看、运行、改名和删除；不支持分享和同步。

### PR-09 隐私与安全

- 延续现有网址防火墙和网页提示词注入过滤。
- 用户主动发送的本地聊天消息可以按现有行为保存；Task、回执、分析、日志和 Skill 不重复保存原始输入。
- 运行时事件和分析不得包含凭证、表单值、Skill 输入值或完整页面正文。
- 错误日志不得输出完整动作参数。
- 登录、密码、支付信息和验证码由用户在网页中处理。

## 非功能需求

- 侧边栏命令在本地 200 毫秒内返回“已接收”或校验错误；模型和网页耗时异步展示。
- 动作在单 Task 内串行执行，状态和外部提交阶段先持久化，再通知界面。
- 用户重新连接时不能看到状态倒退；过期事件必须被忽略。
- 没有通过证据的任务不得显示为已完成。
- 确定性表单和媒体旅程各连续运行 10 次，结果必须一致。

## 成功指标

### 发布闸门

准确率标尺见 `product/003`（对齐美团 Tabbit 披露的 Agent **91.8%** / 网页操作 **70%**）。正式分用中等模型（默认 MiniMax-M3）。

- 两条确定性测试旅程各连续 **10/10** 全部通过（G1/G2）。
- 飞书和 B 站黄金旅程：verified 成功率 **≥91.8%**（n≥50 或操作上 10/10）；探路可用 9/10，**不得**据此宣称已对齐 Tabbit（G3/G4）。
- 长尾网页操作池 verified **≥70%**（对齐对方网页操作线）。
- 测试与 Owner 验收中错误成功声明为 0。
- 未记录批准却执行的外部提交为 0。
- 外部提交各持久化边界的重启注入均不造成重复提交。
- 至少一个 Skill 在更换输入及元素顺序后重新完成。
- 确定性媒体后续指令目标绑定成功率为 10/10。
- 对外「达标」材料满足 G8（分母、n、模型、失败分类可审计）。

### 产品观察指标

- 可验证完成率。
- 每个已完成任务的用户介入次数中位数。
- 后续指令目标绑定成功率。
- Skill 保存率和七日内再次运行率。
- 外部提交的批准、拒绝与结果不确定次数。
- 按产品、模型、网站、登录和环境分类的失败分布。

分析只记录事件类型、耗时、失败类别和计数，不记录用户输入或页面正文。

## 验收标准

1. 在表单测试页中，Agent 能填写多个字段，在提交前停止；批准后只提交一次，观察成功状态并生成回执。
2. 用户拒绝提交后动作不执行，任务仍可继续或取消。
3. 点击提交和按回车提交都必须触发相同批准流程。
4. Planner 声称完成但任一必需条件失败时，界面不得显示已完成。
5. 执行前已经成立、来自旧轮次或错误目标的条件不能证明本轮完成。
6. 用户完成确认必须匹配当前任务、轮次、条件和版本；重复或过期确认不能错误完成任务。
7. B 站旅程中，“暂停这个视频”绑定原媒体目标，并以媒体暂停状态完成验证。
8. 关闭并重开侧边栏后，最后状态和既有回执仍存在；中断任务不得显示为成功。
9. 在外部提交开始后模拟扩展重启，系统显示结果不确定并且不自动再次提交。
10. 同一命令重复到达时只生效一次；批准、取消和后续指令冲突时按当前任务版本串行处理。
11. 已完成表单任务可以保存为 Skill；修改输入和元素顺序后仍能重新完成。
12. 本地存储、界面事件、分析和日志中不出现凭证、表单值、Skill 输入值或完整页面正文。
13. 旧原始动作回放入口被禁用；任何可执行动作都不能绕过统一风险判断。

## 交付顺序

1. 可验证表单提交：任务状态、批准、提交后条件和回执形成最小闭环。
2. 连续媒体控制：后续指令绑定、通用媒体观察和暂停验证。
3. 本地 Skill 重跑：语义模板、输入校验、当前页面重规划和完成验证。
4. 中断恢复与隐私回归：覆盖外部提交各阶段、存储和日志。
5. 真实 Chrome 固定协议验收：飞书与 B 站各 10 次，并发布完整失败分类。

## 假设与停止条件

### 待验证假设

- 现有通用浏览器动作足以覆盖首批表单任务。
- 有限的站点无关完成条件可以覆盖飞书与 B 站，无需专用适配器。
- 用户愿意只为外部提交而介入批准。
- 语义 Skill 比动作回放更能适应页面小改动。
- Owner 会在七日内再次运行至少一个 Skill。

### 停止或降级条件

- 任一真实黄金旅程经过两轮根因修复后仍不超过 5/10：暂停扩展范围并重新评估执行路径。
- 任一外部提交路径可以绕过批准：禁止发布自动提交能力。
- Skill 无法适应元素顺序变化：从本周期移除保存入口，不把动作回放包装成 Skill。
- 中断恢复可能重复外部提交：关闭自动恢复该路径并重新设计。
- 任何持久化记录或日志出现表单值、Skill 输入或凭证：阻止发布并扩大脱敏边界。

## 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 网站结构变化 | 目标定位或验证失败 | 语义重新规划，不依赖旧索引；失败时请求重新绑定 |
| 模型错误判断 | 误报完成或错误动作 | 运行时验证拥有最终完成权；动作统一经过风险判断 |
| 扩展生命周期中断 | 丢失上下文或重复提交 | 先持久化状态；恢复前重观测；不自动重试不确定提交 |
| 批准过多 | 用户重新变成逐步操作员 | 只批准外部提交；读取与可逆动作默认自动执行 |
| 条件类型不足 | 任务频繁等待用户确认 | 记录不支持比例；超过有效任务 20% 后再扩展条件 |
| Agent 只成为模型外壳 | 缺少长期产品价值 | 沉淀任务状态、完成证据、Skill 和真实失败数据，而非只包装聊天界面 |
